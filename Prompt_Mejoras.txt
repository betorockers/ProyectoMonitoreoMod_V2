Actúa bajo el rol de Principal Software Architect y Staff Engineer experto en Windows Desktop Environments, Python Avanzado, Automatización de Navegadores, Seguridad Informática y Empaquetado Hermético de Software (Nuitka/DevOps).

El objetivo es auditar, refactorizar e implementar la estructura de código y configuración de compilación de nivel de producción para una aplicación híbrida (Backend FastAPI + Frontend Next.js SSG). Esta aplicación está diseñada para ejecutarse de forma EXCLUSIVA en sistemas operativos Windows (Windows 10/11 Home, Pro, LTSC o IoT) en equipos limpios/reformateados (sin entornos de desarrollo, sin Python, sin Node.js, sin dependencias externas). La aplicación debe ser completamente autónoma (Standalone / Portable) y empaquetada mediante un instalador nativo de Windows.

Aplica estrictamente principios SOLID, Clean Code, DRY (Don't Repeat Yourself), Type Hinting absoluto (PEP 484), manejo exhaustivo de excepciones y estándares de seguridad de código de nivel militar.

---

### MÓDULO 0: PROTOCOLO DE AUDITORÍA PREREQUISITO (Calidad, Limpieza y Seguridad)
Antes de proceder con cualquier modificación de código, inicialización o preparación de despliegue, realiza de forma obligatoria una revisión completa de la estructura del proyecto desde la raíz. Debes asegurar los siguientes estándares:

1. **Eliminación de Código y Estructuras Huérfanas:** Identifica y elimina de forma absoluta cualquier rastro de código fantasma, funciones obsoletas o sin llamadas de retorno, variables sin uso, así como carpetas huérfanas o vacías que ensucien el árbol del proyecto.
2. **Arquitectura Limpia y Modularización Extrema:** Estructura todo el código en módulos altamente cohesivos y de bajo acoplamiento. Está prohibido el uso de archivos monolíticos gigantescos. Cada componente debe tener una única responsabilidad (Single Responsibility Principle).
3. **Seguridad de Código Nivel Militar:** Sanitiza todas las entradas de datos, valida esquemas rígidamente con Pydantic, evita de forma absoluta la exposición o almacenamiento en duro (hardcoded) de secretos, tokens o credenciales directas, e implementa cifrado/ofuscación donde sea crítico para evitar ingeniería inversa simple.
4. **Auto-Documentación y Comentarios Exhaustivos:** Cada módulo, clase, método y función compleja debe estar 100% comentada y documentada utilizando Docstrings estandarizados (Google Style o Sphinx). Explica con claridad el "por qué" de las implementaciones, no solo el "qué".
5. **Actualización de la Documentación Maestra:** Todos los hallazgos de la auditoría inicial, la arquitectura modular resultante, el mapa de dependencias y los flujos lógicos deben quedar perfectamente asentados y actualizados en la documentación maestra del proyecto.

### MÓDULO 1: CAPA DE INTERFAZ DE USUARIO (UI/UX) - OPERACIONALIDAD 100% REAL
Cada elemento interactivo de la interfaz (botones, inputs, toggles) debe ser completamente funcional en producción y estar libre de efectos secundarios ("side effects") o fallos por falta de permisos en el sistema operativo Windows del cliente.

1. **Interacciones Nativas Blindadas:** Toda funcionalidad del Frontend que interactúe con el sistema anfitrión (como el botón de "Copiar al portapapeles") debe estar implementada con un fallback robusto. Si la API del navegador (`navigator.clipboard.writeText`) encuentra restricciones en entornos de escritorio empaquetados, debe proveer un mecanismo alternativo automático o canalizar la acción de forma segura a través de un endpoint dedicado en el Backend (FastAPI ejecutando código nativo de Windows si es necesario) para asegurar que el contenido copiado (logs, reportes, tokens) pueda ser pegado inmediatamente (`Ctrl + V`) en un Bloc de notas, cuerpo de correo o cualquier software externo de uso común.
2. **Flujos síncronos/asíncronos validados:** Asegura que los botones de acción muestren estados visuales claros (Cargando / Éxito / Error) y que no se congelen ni dejen procesos bloqueados si la operación tarda o falla.

### MÓDULO 2: Arquitectura de Rutas Dinámicas y Servido del Frontend (Nuitka Context Awareness)
Dado que el binario se ejecutará congelado en un directorio arbitrario del cliente, el artefacto debe resolver sus rutas en tiempo de ejecución sin depender del contexto de desarrollo.

1. Diseña un componente de configuración (`config.py`) usando un patrón Singleton o métodos de clase estáticos con `pathlib.Path`.
2. Implementa una lógica infalible para detectar si la aplicación corre compilada (`sys.frozen` o `__compiled__`). Debe usar `sys.executable` para resolver la ruta raíz real del binario ejecutable de Windows (manejando correctamente subcarpetas si se instala en `C:\Program Files`).
3. El frontend de Next.js se exportará previamente como SSG (`output: 'export'` generando la carpeta `out/`). Configura la instancia de FastAPI para montar de forma segura y absoluta esta carpeta de archivos estáticos mediante `StaticFiles(directory=..., html=True)`. Todo el frontend debe quedar embebido dentro del binario final o dentro de la estructura empaquetada e inviolable del instalador.

### MÓDULO 3: Motor de Scraping Invisible y Autónomo (Anti-Bots Windows Bypass)
El scraper debe evadir las heurísticas de Cloudflare/WAF sin levantar ventanas del navegador que interrumpan al usuario común, y sin requerir que el usuario tenga Google Chrome instalado.

1. Diseña un Gestor de Contexto (`Context Manager`) robusto para la inicialización de un navegador automatizado (usando `undetected-chromedriver` o un enfoque nativo de automatización compatible con Nuitka).
2. Implementa una estrategia de "Headed Invisible" nativa para Windows:
   - Inyección de argumentos CLI al driver para renderizado fuera de la pantalla visible (`--window-position=-32000,-32000` y `--window-size=1920,1080`).
   - Ocultamiento absoluto de la consola de comandos de Windows (`cmd.exe`) que levanta el proceso del driver (manejando `creationflags=subprocess.CREATE_NO_WINDOW` en la instanciación interna del proceso hijo).
3. **Aislamiento de Dependencias:** El script debe incluir lógica para descargar de forma automática el binario de Chromium/WebDriver adecuado o buscarlo en una ruta interna relativa al ejecutable de la app, garantizando que funcione en una PC recién formateada sin Chrome.
4. Asegura la destrucción total de procesos (`chromedriver.exe`, `chrome.exe`) en el bloque `finally` para evitar fugas de memoria (Memory Leaks) en la máquina del cliente.

### MÓDULO 4: Pipeline de Compilación Ultra-Hermética con Nuitka
Proporciona el comando de compilación exacto de Nuitka optimizado para generar un ejecutable Windows profesional. El comando debe incluir explícitamente los siguientes flags arquitectónicos:

- `--standalone`: Para incluir todo el entorno de ejecución de Python.
- `--windows-disable-console`: Para ocultar la ventana de comandos negra de Python al iniciar la aplicación (comportamiento de app comercial de Windows).
- `--include-data-dir=out=out`: Para forzar la inclusión de los archivos estáticos de Next.js dentro de la distribución compilada.
- `--plugin-enable=anti-bloat`, `--plugin-enable=pydanticv2`: Optimizaciones de frameworks de datos.
- `--windows-dependency-walker-mode=executable`: Para asegurar que se analicen y empaqueten todas las DLLs de bajo nivel (`vcruntime140.dll`, etc.).
- `--assume-yes-for-downloads`: Para evitar interrupciones de prompts en sistemas CI/CD o scripts automatizados de compilación.

### MÓDULO 5: Empaquetado Enterprise e Instalador Nativo (Target: Inno Setup / MSI)
El resultado `.dist` de Nuitka es una carpeta con múltiples archivos y DLLs. Para el usuario final "común y silvestre", requerimos transformar esto en un único archivo instalador nativo de Windows (ej. `.exe` instalable).

1. Genera la especificación técnica o un script de automatización compatible con **Inno Setup** (archivo de configuración `.iss`) o herramientas similares basadas en CLI de Windows.
2. El script de empaquetado debe:
   - Copiar la carpeta completa de salida de Nuitka.
   - Registrar la aplicación de manera limpia en el sistema de archivos del usuario (`C:\Program Files\TuApp`).
   - Crear accesos directos automáticos en el **Escritorio** y en el **Menú Inicio**.
   - Incluir un desinstalador limpio y estándar de Windows accesible desde el Panel de Control.

Presenta el código completamente estructurado en módulos listos para producción, acompañado de la receta exacta de compilación, las auditorías de seguridad ejecutadas, la validación de eventos UX y la actualización formal de la documentación maestra.
